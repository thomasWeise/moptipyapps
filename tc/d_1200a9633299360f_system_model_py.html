1""" 

2An extended dynamic control problem `Instance` with a model for the dynamics. 

3 

4An :class:`~moptipyapps.dynamic_control.instance.Instance` is a combination of 

5 

6- a set of differential equations 

7 (:mod:`~moptipyapps.dynamic_control.system`) that govern the change 

8 `D=ds/dt` of the state `s` of a dynamic system (based on its current 

9 state `s` and the controller output `c(s)`), i.e., `ds/dt=D(s,c(s))` and 

10- a blueprint `c(s, p)` of the controller `c(s)` 

11 (:mod:`~moptipyapps.dynamic_control.controller`). 

12 

13The blueprint of the controller is basically a function that can be 

14parameterized, so it is actually a function `c(s, p)` and the goal of 

15optimization is to parameterize it in such a way that the figure of merit, 

16i.e., the objective value (:mod:`~moptipyapps.dynamic_control.objective`) 

17of the system (usually the sum of squared state values and squared controller 

18outputs) is minimized. Parameterization means finding good values `p` such 

19that the above goal is reached. In other words, we want to synthesize a 

20controller (by finding good values of `p`) in such a way that the state 

21equations drive the system into a stable state, usually ideally the origin of 

22the coordinate system. 

23 

24Now here this :class:`~moptipyapps.dynamic_control.instance.Instance` is 

25extended to a :class:`~moptipyapps.dynamic_control.system_model.SystemModel` 

26by adding a parameterized model `M(s, c(s), q)` to the mix. The idea is to 

27develop the parameterization `q` of the model `M` that can replace the actual 

28system equations `D`. Such a model receives as input the current system state 

29vector `s` and the controller output `c(s)` for the state vector `s`. It will 

30return the differential `D` of the system state, i.e., `ds/dt`. In other 

31words, a properly constructed model can replace the `equations` parameter in 

32the ODE integrator :func:`~moptipyapps.dynamic_control.ode.run_ode`. The 

33input used for training is provided by 

34:func:`~moptipyapps.dynamic_control.ode.diff_from_ode`. 

35 

36What we do here is to re-use the same function models as used in controllers 

37(:mod:`~moptipyapps.dynamic_control.controller`) and learn their 

38parameterizations from the observed data. If successful, we can wrap 

39everything together into a `Callable` and plug it into the system instead of 

40the original equations. 

41 

42The thing that :class:`SystemModel` offers is thus a blueprint of the model 

43`M`. Obviously, we can conceive many different such blueprints. We could have 

44a linear model, a quadratic model, or maybe neural network (in which case, we 

45need to decide about the number of layers and layer sizes). So an instance of 

46this surrogate model based approach has an equation system, a controller 

47blueprint, and a model blueprint. 

48 

49An example implementation of the concept of synthesizing models for a dynamic 

50system in order to synthesize controllers is given as the 

51:mod:`~moptipyapps.dynamic_control.surrogate_optimizer` algorithm. 

52Examples for different dynamic systems controllers (which we here also use to 

53model the systems themselves) are given in package 

54:mod:`~moptipyapps.dynamic_control.controllers`. 

55 

56The starting points of the work here were conversations with Prof. Dr. Bernd 

57NOACK and Guy Yoslan CORNEJO MACEDA of the Harbin Institute of Technology in 

58Shenzhen, China (哈尔滨工业大学(深圳)). 

59"""
